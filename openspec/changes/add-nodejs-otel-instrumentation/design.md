# Design

## Context

Node.js OTel instrumentation (`otel_nodejs.go`) currently exists as a print-only stub: it detects projects and entrypoints but only prints manual instructions.

Current Node.js stub flow: detect `package.json` projects → user selects one → print `npm install` and `node --require` commands. No packages are installed, no processes are launched, no `.otel/` directory is created.

The existing shared infrastructure (`otel_runtime_scan.go`, `otel_process.go`, `otel_env.go`) provides project scanning, process management, and env var generation that can be reused directly.

## Goals / Non-Goals

**Goals:**

- Detect Node.js projects including monorepos (via `workspaces` field), Next.js projects (via `next.config.*` or `next` in deps), and Nuxt projects (via `nuxt.config.*` or `nuxt` in deps)
- Auto-detect the project's package manager from lockfiles (npm/yarn/pnpm)
- Create a `.otel/` directory inside the project with its own `package.json` for OTel deps — never modify the app's `package.json`
- Install OTel packages by running `npm install` inside `.otel/`
- Launch instrumented apps: regular apps via `node --require` from `.otel/` dir, Next.js/Nuxt via framework-specific wrapper scripts in `.otel/`
- Register `dtwiz install otel-node` as a standalone command
- Extend `dtwiz uninstall otel` to clean up `.otel/` directories and instrumented processes
- Support `--dry-run` for the install command

**Non-Goals:**

- Changing the app's `package.json` or `node_modules`
- Using `NODE_OPTIONS` environment variable
- TypeScript-specific runners (`tsx`, `ts-node`) — deferred to follow-up
- Enabling Node.js by default in the combined `dtwiz install otel` flow
- A separate `dtwiz uninstall otel-node` command

## Decisions

### 1. Isolated `.otel/` directory for OTel deps

OTel packages are installed in a `.otel/` directory inside the project root, not in the project's own `node_modules`. This means:

- The app's `package.json` is never touched
- No risk of version conflicts with the app's dependencies
- Node.js resolves `require()` relative to the file that calls it, so `--require .otel/node_modules/@opentelemetry/auto-instrumentations-node/register` works when running from the `.otel/` directory
- Clean uninstall: just `rm -rf .otel/`

The `.otel/package.json` is generated by dtwiz with the OTel packages as dependencies. `npm install` is always used inside `.otel/` regardless of the project's package manager — this simplifies the implementation since npm is universally available where Node.js is installed.

**Alternative considered:** Installing OTel packages directly in the project's `node_modules`. Rejected because it modifies the user's dependency tree, can cause version conflicts, and makes uninstall messy.

**Alternative considered:** Using a global npm install. Rejected because global installs require elevated permissions and affect all projects on the machine.

### 2. Framework bootstrap scripts (Next.js, Nuxt)

Framework projects need framework-specific launch behavior, so dtwiz generates bootstrap scripts in `.otel/`.

**Next.js:** dtwiz generates `.otel/next-otel-bootstrap.js` and launches the app with:
`node .otel/next-otel-bootstrap.js start`

The wrapper loads `@opentelemetry/auto-instrumentations-node/register`, then delegates to `next/dist/bin/next`. OTEL\_\* env vars are passed via `cmd.Env` at launch time — they are not embedded in the script, avoiding writing secrets to disk.

**Nuxt:** dtwiz generates `.otel/nuxt-otel-bootstrap.mjs` and launches the built Nitro server with:
`node --import .otel/nuxt-otel-bootstrap.mjs .output/server/index.mjs`

Nuxt does **not** delegate to the Nuxt CLI in this flow. The ESM bootstrap uses `module.register()` to install loader hooks before loading OTel registration so instrumentation applies to Nitro's ESM runtime.

Regular Node.js apps don't need wrapper scripts — they use `node --require @opentelemetry/auto-instrumentations-node/register <entrypoint>` directly, with OTEL\_\* env vars set on the process.

**Alternative considered:** Using `NODE_OPTIONS='--require ...'`. Works but is an env var modification the user must maintain. The direct `--require` approach is contained within the dtwiz-managed command.

### 3. Package manager detection from lockfiles

The project's package manager is detected by checking for lockfiles:

| Lockfile            | Package Manager |
| ------------------- | --------------- |
| `package-lock.json` | npm             |
| `yarn.lock`         | yarn            |
| `pnpm-lock.yaml`    | pnpm            |
| (none found)        | npm (default)   |

This is informational — displayed in the plan preview so the user knows their project was correctly identified. The `.otel/` directory always uses `npm` since it's universally available.

### 4. Monorepo detection via `workspaces`

When `package.json` contains a `"workspaces"` field (array of globs or object with `packages` array), the project is identified as a monorepo. Each workspace directory containing its own `package.json` is treated as an individual `ScannedProject`. This feeds into the existing `scanProjectDirs()` mechanism.

The monorepo root itself is also listed as a project option, giving the user the choice to instrument at the root level or a specific workspace package.

### 5. Node.js-specific env vars

`generateOtelNodeEnvVars()` extends `generateBaseOtelEnvVars()` with:

- `OTEL_NODE_RESOURCE_DETECTORS=all` — recommended by Dynatrace for comprehensive resource detection

This function generates the env vars map that is passed to the spawned process via `cmd.Env`.

### 6. Running regular apps from `.otel/` directory

For regular Node.js apps, the command is:

```bash
node --require @opentelemetry/auto-instrumentations-node/register <entrypoint>
```

This runs with `CWD = .otel/` so that `require()` resolves `@opentelemetry/auto-instrumentations-node` from `.otel/node_modules/`. The entrypoint path is made relative to `.otel/` (e.g., `../server.js`).

### 7. Uninstall integrated into `dtwiz uninstall otel`

Node.js cleanup is added to the existing `UninstallOtelCollector()` rather than creating a separate command. The uninstall flow:

1. Finds `.otel/` directories by scanning CWD and common dev directories for `.otel/package.json` containing `@opentelemetry`
2. Finds running `node` processes whose command line includes `@opentelemetry/auto-instrumentations-node/register`, `.otel/next-otel-bootstrap.js`, or `.otel/nuxt-otel-bootstrap.mjs`
3. Adds a "Node.js instrumentation" subsection to the existing preview
4. On confirmation: kills instrumented processes, removes `.otel/` directories

The existing collector uninstall stays as-is — Node.js cleanup is additive.

### 8. Cobra command following existing patterns

`installOtelNodeCmd` is registered under `installCmd` with `Use: "otel-node"` and `Args: cobra.NoArgs`. It gets a `--service-name` flag (same as Python and Java). The parent's `--dry-run` persistent flag is inherited. No separate uninstall command — cleanup is part of `uninstall otel`.

## File Layout

| File                | Responsibility                                                                                                                                                                                                                                                              |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `otel_nodejs.go`    | Project/monorepo/Next.js/Nuxt detection, package manager detection, `NodeInstrumentationPlan`, `buildNodeInstrumentationPlan`, `DetectNodePlan`, `InstallOtelNode`, `Execute`, `PrintPlanSteps`, `.otel/` directory creation, wrapper script generation, env var generation |
| `otel_uninstall.go` | Extended with `findNodeOtelDirs()`, `findInstrumentedNodeProcesses()`, Node.js section in `UninstallOtelCollector()`                                                                                                                                                        |
| `otel.go`           | Updated `createRuntimePlan()` Node.js branch to pass `envURL` and `platformToken`                                                                                                                                                                                           |
| `cmd/install.go`    | `installOtelNodeCmd` registration                                                                                                                                                                                                                                           |

All changes are in the existing `installer` package — no new packages or public API changes beyond `InstallOtelNode()`.

## Risks / Trade-offs

- **[npm availability]** → `.otel/` always uses npm. If a user only has yarn/pnpm installed without npm, `npm install` will fail. Mitigation: npm is bundled with Node.js by default; if missing, the prerequisite check will catch it.
- **[Next.js internal path]** → `./node_modules/next/dist/bin/next` is an internal path that could change across Next.js versions. Mitigation: this is the stable entry point used by the `next` CLI itself; if it changes, the error will be obvious and the path can be updated.
- **[Nuxt internal path]** → `./node_modules/nuxt/bin/nuxt.mjs` is an internal path that could change across Nuxt versions. Same mitigation as Next.js — this is the stable entry point used by the `nuxt` CLI.
- **[CWD trick for regular apps]** → Running from `.otel/` with a `../` entrypoint path works for simple cases but may break if the app uses `__dirname`-relative paths that depend on CWD being the project root. Mitigation: set the `--require` path to the absolute path of the module in `.otel/node_modules/` so CWD can remain the project root.
- **[Monorepo workspace glob resolution]** → Complex glob patterns in `workspaces` field may not be fully resolved. Mitigation: handle simple array of directory names/globs; complex patterns fall back to existing `scanProjectDirs()` which already scans subdirectories.
